    Run 1: The Forecast Intelligence Platform¶

    What You're Starting With

    From Lift 1, your team has:

    • A verified deployment pipeline: you deployed a landing page and confirmed continuous delivery works
    • A context architecture discussion: what goes in the project root vs. path-scoped rules vs. subdirectory context
    • An outcome gap analysis: which missing results on your team could be addressed by skills encoding accountability for those outcomes
    • A visibility problem framing: the question isn't "does the code run" but "does the AI-generated analysis accurately reflect the published forecast"

    Your repository has pre-seeded data for all 9 UAC zones: forecast snapshots, NWS weather data, SNOTEL snowpack readings, zone configurations, and alert thresholds. API documentation covers all five data sources for live integration.

    The Challenge¶

    Build a multi-zone forecast intelligence platform with an alert pipeline connecting two user groups. The deterministic pipeline ingests published forecasts, weather, and snowpack data across UAC zones, parses danger ratings, and routes alerts through configurable thresholds. The AI-powered analysis layer calls the Anthropic API to generate the content of those alerts: translating the forecaster's expert assessment into plain language that operations staff and backcountry travelers can act on. The deterministic layer decides when to alert. The AI layer decides what the alert says.

    You have the Director's Toolkit from Lift 1: a verified deployment pipeline, a context architecture for managing complexity across subsystems, skills that encode your team's role gaps as AI roles, and delegation patterns for running parallel workstreams. Put all of it to work.

    Your system should feel like something an operations manager would actually open at the start of each day, or a backcountry skier would check before a trip. Not a code demo. A tool.

    Who You're Building For

    Forecast operations staff: people who manage the UAC forecasting program across all 9 zones. They don't write the daily forecast; the zone forecasters do that. Operations staff care about the big picture: which zones have the highest danger right now, are any zones missing data, where should field resources be deployed today, what patterns are emerging across the range. Their pain point is the cognitive load of monitoring 9 zones x 3 data sources without a synthesized view.

    Backcountry recreationists: skiers, snowboarders, and mountaineers planning a trip into the backcountry. They check the forecast before heading out but may not have the expertise to parse a technical forecast discussion. When the forecast says "Persistent Slab problem on NW-N-NE aspects above 9,500 feet," the backcountry skier needs to hear "Avoid north-facing slopes above 9,500 feet today." That translation, from expert terminology into actionable trip-planning language, is the core value your AI layer provides.

    The forecasters who publish the daily danger ratings are the domain experts whose assessments are your system's ground truth. Their published forecasts are the authoritative input. Your system makes that expertise accessible to the people downstream.

    How You're Working¶

    Teams of four, one deliverable. How you organize is your call. There are two decisions to make:

    How you divide the work:

    • Divide and conquer: each person (or pair) takes a different component. One handles data ingestion, another builds the AI alert component, another works on the dashboard. The goal is coverage: build known components in parallel and integrate.
    • Fan-out / fan-in: each person takes the same brief and builds independently, then the team converges to synthesize the best of what everyone found. The goal is divergence: four people asking different questions of their agents explore the solution space far more widely than one group ever could. This is most valuable in unknown domains. If calling an LLM from your code is unfamiliar territory, fan out to discover viable approaches, then fan in to decide.

    How you group people:

    • Mobbing (whole team, one screen): best for foundational decisions where shared context matters. Architecture choices, context structure, alert pipeline design. Also the right mode for fan-in: converge around one screen, ask Claude to analyze the branches, synthesize the best into main.
    • Pairing (two and two): best when you have parallel swim lanes that benefit from shared context within each pair. Each pair keeps each other honest.
    • Solo: best for maximum parallelism when the work is well-defined or when you're deliberately fanning out for divergent exploration.

    These compose. You might mob to agree on architecture, then divide and conquer in pairs for implementation, then mob again to integrate. Or fan out solo to explore the AI alert component, then mob to synthesize. Match the pattern to the moment.

    What to Build¶

    Items are listed in priority order. If time is tight, focus on the items near the top first.

    • At least one AI-powered alert component. The system calls the Anthropic API at runtime to generate the content of at least one alert type: a plain-language translation of the forecast for backcountry travelers, a contextual explanation of why an alert fired for operations staff, or risk factor extraction from the forecast discussion text. This is the layer that will need evaluation in Run 2. Build it now so you have something to measure. Prioritize this. Everything in Session 2 depends on having an AI layer to evaluate.
    • Multi-zone data ingestion and display. The engine loads forecast data for multiple UAC zones from the pre-seeded snapshot and displays current conditions on a dashboard. An operations manager opening the dashboard sees danger ratings, active avalanche problems, and zone-level summaries without clicking into each zone individually.
    • Continuous delivery from the start. The deployment pipeline is verified and you're shipping to your live URL throughout the run, not just at the end. Every significant feature goes live as soon as it works.
    • Context architecture implemented. Your project has a root-level context file using the bootstrap pattern, plus at least one path-scoped rule or subdirectory context for a specific subsystem (data ingestion, analysis, dashboard, tests, whichever your team decides warrants scoped context).
    • At least one skill filling an outcome gap. Your team identified missing outcomes in Lift 1's discussion. Turn at least one into a working skill: decomposition (PM), test generation (QA), UI component conventions (designer), or code review patterns (tech lead). The skill is version-controlled and used during the run.

    These are options for teams that finish the baseline capabilities. Your team can also define your own stretch goals based on what interests you.

    • Cross-source data correlation. Combine forecast, weather, and snowpack data to surface relationships: a zone's danger level increased and its SNOTEL station shows a foot of new snow in 24 hours, or temperature is rising and the forecast mentions warming as a concern. Your system connects the dots across data sources so operations staff don't have to.
    • Configurable alert engine. Alerts fire based on configurable thresholds: danger level at or above a certain value, danger increased since yesterday, multiple zones showing elevated conditions simultaneously. Use the pre-seeded alert threshold configuration as a starting point, with the ability to adjust rules.
    • AI-powered zone analysis. Use the Anthropic API to generate a richer plain-language interpretation for individual zones, translating the published danger rating, weather context, and snowpack trends into language a backcountry skier or operations manager can act on without domain expertise.
    • Deep fan-out. Each team member builds the AI alert component independently on their own branch before converging. Compare implementations: what did different agent teams produce when given the same brief? This is where fan-out/fan-in shines, using non-determinism to explore the solution space.
    • Live API integration. Supplement pre-seeded data with live API calls for real-time conditions, with caching to protect the public APIs.

    Share what you've built! Proud of what your team created? Post a screenshot or a quick description in #impact-lab-show-and-tell on Slack and show the whole conference what you're building.

    Tips

    • Deploy first, build second. You verified the pipeline in Lift 1. Keep using it. The first feature you build should be live before you start the second. Continuous delivery means continuous, not "deploy at the end."
    • Your context architecture matters immediately. This is a multi-subsystem project. A monolithic context file will degrade as the project grows. Set up your layered architecture before the codebase gets complex; it's easier to scope context to subsystems when the subsystems are small.
    • Build against the snapshot, not the APIs. The multi-zone snapshot contains everything you need for Run 1. Live APIs add latency, rate limit risk, and failure modes you don't want to debug during a build sprint. Wire in live data as a stretch goal once the engine works against static data.
    • Your AI layer translates; it doesn't analyze. The forecaster is the domain expert. Your system's job is to take their published assessment and make it accessible, not to do independent avalanche science. When the forecast says "Persistent Slab on NW-N-NE aspects above 9,500 feet," the AI should translate that into "Avoid north-facing slopes above 9,500 feet today." The forecaster's judgment is always the ground truth.
    • Use Claude to learn the Anthropic API. If calling an LLM from your code is new territory, don't stop to read documentation yourself. Paste the API docs directly into your conversation and ask Claude to walk you through the pattern: show me an API call, explain the response structure, help me understand how to pass context. Claude can learn a new API from its own documentation faster than you can read it. Keep going until you feel ready to implement. That's the research skill in action.
    • The golden datasets are your future answer key. You won't build eval harnesses until Run 2, but knowing the golden datasets exist should shape how you build now. The expected outputs (danger ratings, avalanche problems, alert decisions) serve double duty: direct comparison targets for testing the deterministic pipeline, and rubric criteria for evaluating whether your AI-generated analysis accurately reflects what the forecaster identified.
